Section 1

Executive Summary

SOL Webhook currently mixes tenant/property facts across JSON packages and hard-coded runtime logic. Supabase should become the database of record for tenant identity, property data, venue catalogs, routing teams, templates, and operational events. The webhook should consume a stable runtime config derived from that data, while LOGS should read operational event tables and views.

GoalOne source of truth

Reduce duplicated config and hard-coded tenant details.

MethodSupabase schema contract

Define ownership before implementation.

RuntimeCompiled config view

Webhook reads stable config, not scattered raw tables.

ReviewCommentable artifact

Every major target can receive feedback.

Section 2

Objectives

  • Create a shared Supabase schema that both Property Admin and LOGS can trust.
  • Move tenant/property-specific facts out of runtime code and into managed configuration.
  • Preserve existing webhook behavior until schema, migration, and runtime consumption are approved.
  • Separate deployment secrets from tenant-editable settings and operational event data.
  • Give reviewers a commentable, exportable, single-page artifact for schema feedback.

Section 3

Definitions

Tenant

The client account or organization using SOL.

Property

A hotel, casino, venue, or site owned by a tenant.

Venue

A restaurant, bar, room service, entertainment location, or guest-facing service point.

Runtime Config

The stable data shape consumed by webhook and Fast Path logic.

Template

Localized guest-facing response text with variables.

Operational Event

A logged inbound, outbound, status, reservation, complaint, or middleware event.

Hard-Code Replacement

A current literal value that should move into env, tenant config, or runtime config.

Review Target

A stable ID that comments attach to, such as table.venues.

Section 11

MVP Scope

  • MVP includes tenant, property, settings, teams, venues, aliases, templates, runtime flags, messages, reservations, and operational events.
  • MVP does not require a live property admin UI connected to Supabase.
  • MVP does not require the webhook to read raw tables directly.
  • MVP should seed current demo data into the schema for validation.
  • MVP must preserve current webhook behavior until runtime config migration is approved.
